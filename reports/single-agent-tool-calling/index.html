Single Agent / Tool Calling

從純 RAG 到工具導向 Agent

這份筆記說明如何從「所有問題都先查知識庫」的 RAG pipeline，演進成一個能依照需求選擇工具的 single orchestrator agent。它是 LINE OA Agentic Workflow 的第一階段技術地基。

一句話總結

RAG 不應該是固定主流程，而應該是工具清單中的一個 tool。第一版 agentic system 最穩的做法，是一個 Orchestrator Agent 搭配多個後端 tools，而不是一開始就拆成多個獨立 agents。

為什麼純 RAG 不夠

RAG 擅長靜態知識

產品規格、FAQ、SOP、政策文件、how-to 教學，很適合透過 embedding 和 vector search 檢索後回答。

RAG 不擅長即時動作

庫存、預約時段、購物車、訂單、客戶狀態、ticket 建立，這些不是「文件答案」，而是要呼叫工具或 API。

目標架構：Single Orchestrator + Tools

User message → Single Orchestrator Agent → searchKnowledgeBase → lookupCustomer → searchProducts → addToCart → createBooking → createHumanTicket → writeAuditEvent → Final response

模型負責判斷要用哪個工具；後端負責驗證、執行、記錄與回傳結果。模型不能直接碰 database、金鑰、付款或第三方 API。

RAG 只是其中一個工具

Tool用途是否需要確認
searchKnowledgeBase查 SOP、FAQ、產品文件、政策否
searchProducts查商品、規格、價格、庫存否
addToCart把確認過的商品加入購物車是
createBooking建立預約或保留時段是
createHumanTicket轉真人客服，建立 ticket視情境

Agent Loop

基本流程

  • 把對話和工具定義送給模型。
  • 模型選擇回答或要求 tool call。
  • 後端驗證並執行 tool。
  • 把 tool result 送回模型產生最終回覆。

必要限制

  • max steps。
  • timeout。
  • tool allowlist。
  • audit log。
  • confirmation gate。

安全設計

模型不能直接做的事

  • 不能直接寫 database。
  • 不能自行產生 SQL 後執行。
  • 不能直接付款、退款、取消訂單。
  • 不能自行讀取 secret。

後端必須做的事

  • 驗證 tool input。
  • 檢查權限與 scope。
  • 記錄 tool calls 和 audit events。
  • 高風險操作要求使用者確認。

落地順序

Phase內容
1建立 webhook / chat endpoint、message log、workflow state。
2加入 searchKnowledgeBase tool。
3加入一個 action tool，例如 add to cart 或 booking。
4加入 confirmation gate、human handoff、audit log。
5工具變多或風險變高後，再評估 multi-agent。
