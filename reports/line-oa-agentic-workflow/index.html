LINE OA + Agentic Workflow

讓 LINE 客服從「回答問題」進化到「完成任務」

這個架構把 LINE Official Account 當成客戶入口，由後端 Orchestrator 管理對話狀態、RAG 知識查詢、交易工具、預約流程與真人接手。LLM 負責理解自然語言，真正的操作則由受控的 tools 執行。

Core Architecture

不是讓 AI 自由操作，而是讓 AI 在流程裡做判斷

最穩的方式是讓 Orchestrator 掌握狀態與規則：AI 可以理解問題、選擇工具、整理回覆；但加入購物車、建立預約、購票或轉真人，都必須經過後端 workflow。

1客戶入口
LINE Official Account 接收客戶訊息，透過 Messaging API webhook 送到後端。
Webhook Backend 驗證 signature、記錄 message、載入 conversation state。
→
2Agent Team
S
Supervisor Agent判斷 intent、routing、缺少資訊與是否轉真人。
K
Knowledge Agent查 SOP、FAQ、規範、產品文件，產生有依據的回答。
T
Transaction Agent呼叫購物車、預約、票務、訂單等受控 tools。
→
3受控執行

LINE Reply API 回覆客戶結果，並保留 audit log。

Agent Team

三個 Agent 的邊界要清楚，系統才不會失控

真正可靠的 agent team 不是每個 agent 都什麼都能做，而是每個角色只負責自己的判斷範圍，並且只能使用被允許的 tools。

Supervisor Agent

總控

負責理解客戶意圖、選擇流程、管理 conversation state，並判斷是否要追問、轉真人或交給其他 agent。

  • 判斷 `faq_question`、`add_to_cart`、`booking`、`human_handoff`
  • 管理目前步驟：收集資訊、等待確認、執行、完成
  • 高風險或低信心時停止自動回答

Knowledge Agent

RAG

只負責查已核准的內部文件，例如 SOP、產品規格、FAQ、政策與規範，並回傳可引用的答案依據。

  • 用 metadata route 到正確文件集合
  • 找不到資料時回報 `insufficient_context`
  • 避免憑空承諾退款、保固、價格或法律條款

Transaction Agent

Action

負責呼叫實際業務工具，例如商品查詢、加入購物車、預約球場、保留票券與建立訂單。

  • 只能呼叫明確定義的 backend tools
  • 任何狀態變更都要通過 confirmation gate
  • 工具失敗或付款相關問題轉真人

Implementation Stance

第一版不要真的做成 multiple agents，而是 single orchestrator + tools

這份文件使用 Supervisor、Knowledge、Transaction 三個名稱，是為了說清楚責任邊界；但第一版實作不建議直接做成三個獨立 agent runtime。比較穩的落地方式是先用一個 Orchestrator Agent 控制流程，下面掛多個受控 tools。

Recommended MVP

產品上看起來像 agent team，工程上先做單一 Orchestrator

對使用者來說，系統能查文件、查商品、加入購物車、預約、轉真人，像是一個有多種能力的 AI assistant。對工程來說，第一版應該是一個主控 agent / workflow engine，負責 routing、狀態機、確認閘門與 handoff，再呼叫 RAG、交易、ticket 等 tools。

1 Single Orchestrator
判斷 intent、管理 workflow state、決定下一步。
2 Tools First
RAG、商品、購物車、預約、ticket 都先做成 tools。
3 Multi-agent-ready
介面預留未來拆成 Knowledge / Transaction / Support agents。

Phase 1：Single agent + tools

一個 Orchestrator Agent 搭配 hard rules、lightweight classifier、RAG tool、one action tool、handoff tool。這是外包開發的第一版主規格。

  • 速度較快，debug 較容易。
  • 不會一開始就產生多 agent 溝通成本。
  • 適合先驗證 LINE OA 使用者體驗。

Phase 2：更多 tools 與流程

擴充 booking、cart、order lookup、customer profile、notification、audit。仍可維持單一 Orchestrator，只要 tools schema 與 workflow state 設計清楚。

  • 每個 tool 都有 schema、權限、audit log。
  • 高風險 tool 一律走 confirmation gate。
  • 真人接手透過 ticket / case management。

Phase 3：需要時才拆 multi-agent

當工具太多、選錯率上升、不同任務需要不同 system prompt 或安全邊界，才把模組拆成真正獨立的 agents。

  • Knowledge Agent：低風險文件問答。
  • Transaction Agent：交易 / 預約工具。
  • Support Agent：真人接手與案件摘要。

給外包開發的正確說法

不要一開始寫「請開發 multi-agent 系統」。比較準確的規格是：「請開發 LINE OA Agentic Workflow 系統；第一版採用 single orchestrator agent + tool-calling architecture，並預留未來拆成 multi-agent 的介面。」

Related Technical Background

下面兩份同 repo 的背景文件是這個 LINE OA 外包案的技術地基：單一 agent 文件說明 tool calling / agent loop；multi-agent 文件說明何時需要拆成多 agent。

單一 Agent 架構：從純 RAG 到工具導向 Agent 對應本案 Phase 1：RAG 是一個 tool，不是固定主流程；LLM 透過 tool calling 選擇查文件或執行工具。 多 Agent 架構：何時拆、怎麼拆 對應本案 Phase 3：當工具太多、風險邊界不同或需要專長分工時，再拆成真正的 multi-agent。

Technical Architecture

以 Vercel 部署 Next.js 後台與 Webhook，再把資料層交給 Supabase

這不是純前端網站。Vercel 適合負責 Next.js web app、LINE webhook、API routes、客服後台與短時間 orchestration；Supabase 負責 Postgres、pgvector、Storage、Auth；長時間工作則交給 queue 或 background jobs。

LINE OA / Messaging API → Vercel /api/line/webhook → Orchestrator Service on Next.js → Router / Workflow State → RAG Retriever → Tool Executor → Human Handoff → Supabase Postgres / pgvector / Storage → Queue or Background Jobs for slow tasks → LINE Reply or Push API

Vercel Web / API Layer：必要

LINE 會把使用者訊息送到 webhook，所以一定要有公開 HTTPS endpoint。Vercel 可以部署 Next.js Route Handler，負責驗證 LINE signature、儲存訊息、呼叫 Orchestrator、保護 LLM API key，並透過 LINE Reply / Push API 回覆。

  • 推薦：Vercel + Next.js Route Handler / API Routes。
  • 適合：webhook、admin dashboard、短時間 RAG retrieval、建立 ticket。
  • 注意：不要在 Vercel request 裡跑大型 PDF parsing 或大量 embedding。

主資料庫：Conversation / Ticket / Audit

需要儲存 LINE user、conversation state、messages、tickets、tool calls、audit events。這些是交易與客服流程資料，不適合只放在向量資料庫。

  • 推薦：Supabase Postgres。
  • 可選：Neon Postgres、Railway Postgres、PlanetScale。
  • 重點：要支援 transaction、關聯查詢、權限控管。

Vector DB：RAG 檢索層

文件不應該每次問答時才丟給模型，而是預先 chunk、embedding、寫入 vector index。使用者提問時，先檢索相關 chunks，再交給 LLM 回答。

  • 推薦：Supabase pgvector，跟主資料庫整合最簡單。
  • 可選：Qdrant、Pinecone、Weaviate。
  • MVP：pgvector 足夠，正式大量文件再評估獨立 vector DB。

File Storage：原始文件保存

SOP、PDF、Word、Markdown、Notion 匯出檔需要保存原始檔，方便重新 indexing、版本控管與人工審核。vector chunks 不是原始文件的替代品。

  • 推薦：Supabase Storage。
  • 可選：S3、Cloudflare R2、Google Cloud Storage。
  • 需要 metadata：部門、分類、版本、狀態、visibility。

Queue / Job：補上 Vercel 不適合長任務的部分

Vercel 很適合 webhook 和短 API，但不應把文件抽文字、chunking、embedding、同步 Notion、重新索引、長時間工具執行卡在 request 裡。Webhook 應快速回應，再把慢工作丟到 queue。

  • MVP：Vercel Cron + Supabase Edge Function。
  • 推薦搭配：QStash，適合 Vercel serverless queue。
  • 進階：Cloudflare Queues、BullMQ + Redis。
  • 重點：LINE 使用者要先得到「我正在處理」的狀態回覆。

Admin UI：文件與客服後台

團隊要上傳 SOP、看 conversation、處理 ticket、人工回覆 LINE、看 audit timeline。沒有後台，真人接手和文件維護會很快失控。

  • 推薦：Next.js admin dashboard。
  • 可快速起步：Supabase Auth + 自建簡單列表頁。
  • 未來可接：Zendesk、Intercom、Freshdesk、HubSpot。
架構選項 適合情境 優點 限制
Vercel + Next.js + Supabase + OpenAI MVP 與中小型正式產品 開發快、GitHub 自動部署、前後台同 repo、Postgres + pgvector + Storage 一起解 長任務要丟 queue；大量文件處理不要卡在 Vercel function
Cloudflare Workers + Supabase / R2 重視低延遲與全球部署 Webhook 反應快、成本低、部署簡單 複雜後台與 Node 套件相容性要注意
NestJS / FastAPI + Postgres + Qdrant 企業級、複雜 workflow、多人團隊 服務邊界清楚、適合複雜權限與背景任務 初期工程量較大，不適合快速 PoC
串接現成客服平台 已經有客服流程或 SLA 要求 ticket、指派、客服報表現成 AI workflow、RAG、tool calling 仍需要自建中介層

推薦 MVP 技術組合

以你的常用 stack 來看，第一版建議用 Vercel + Next.js + Supabase + pgvector + Supabase Storage + OpenAI / Claude。這可以最快做出 LINE webhook、RAG、ticket、audit log 與簡單 admin dashboard。

不要省掉的技術元件

  • Webhook signature 驗證：避免偽造 LINE request。
  • Workflow state：避免每次訊息都重新理解上下文。
  • Audit events：追查 AI 判斷、tool call、真人回覆。
  • Queue / background job：避免慢任務拖垮 LINE 回應速度。

Routing Strategy

把「問答」和「動作」分開，讓每條路徑都有規則

Agentic workflow 最容易出問題的地方，是把所有訊息都丟給同一個模型自由處理。比較穩的做法是 hard rules 優先，再用 LLM classifier 判斷 intent。

客戶意圖 Routing 需要的工具 確認規則
詢問 SOP、FAQ、產品規格 Knowledge Agent RAG search、document citation 不需要確認，但要標記資料來源
查商品、規格、庫存、價格 Transaction Agent product search、inventory API 只查詢不需確認
加入購物車、建立預約、保留票券 Transaction Agent + Workflow cart API、booking API、ticket API 執行前必須回 LINE 要求明確確認
付款、退款、取消訂單、客訴 Human Handoff ticket、notification、agent dashboard 預設轉真人，不讓 AI 自行決定
問題不清楚或連續失敗 Supervisor Agent clarifying question、conversation state 最多追問 1 到 2 次，再轉真人

Human Handoff & Audit

真人接手不是把客服拉進 LINE，而是把對話切到受控客服流程

客戶端仍然只看到 LINE OA。內部則由系統建立 ticket、通知客服、把該 conversation / session 狀態改成人工接手，並停止該 session 的 AI 自動回覆。其他客戶或其他新 session 仍可照常走預設 AI 客服。Audit log 則放在後台的稽核紀錄，用來還原「AI 為什麼這樣判斷、誰接手、哪個工具被執行」。

1

觸發轉真人

Supervisor 偵測到退款、客訴、法律、合約、付款失敗、低信心回答，或客戶要求「我要找真人」。

2

先回覆客戶

LINE OA 回：「這個問題我幫你轉給客服專員，請稍候。」讓客戶知道 AI 已停止自行處理。

3

建立 ticket 並指派

系統建立客服案件，附上對話摘要、轉接原因、建議部門、客戶資訊與 AI 已查過的文件。

4

AI 暫停自動回覆

`conversation.status` 從 `ai_active` 改成 `human_pending` 或 `human_active`。這個狀態只鎖定該 session；後續同一案件的訊息只通知客服，不再讓 AI 回覆。其他使用者或其他未轉接的 session 仍照常由 AI 處理。

5

客服從後台回 LINE

真人在客服後台輸入回覆，系統用 LINE Reply / Push API 以 OA 身分回給客戶；結案後再切回 AI。

客服後台看到的案件

human_pending
客戶訊息 「我要退款，這個商品跟說明不一樣。」
轉接原因 退款 + 客訴，高風險，AI 不應自行承諾處理結果。
建議部門 客服 Team，必要時轉財務或主管審核。
AI 摘要 客戶對商品描述不滿，要求退款；尚未確認訂單編號與購買日期。
下一步 客服要求訂單編號、照片或購買紀錄，再依退款 SOP 處理。

Session-scoped Handoff

轉真人不是把整個 LINE OA 的 AI 關掉，而是把「某個使用者的某個 open conversation」切到真人狀態。Webhook 每次收到訊息時，先查 `line_user_id` 是否有未結案 session；如果該 session 是 `human_active`，就只寫入 ticket 並通知客服。如果沒有 open human session，才進入預設 AI routing。

  • 同一個 handoff session：AI 停止自動回覆，訊息進客服後台。
  • 其他新客戶：照常進 AI 客服流程。
  • 同一位客戶的新問題：依規則決定沿用真人 session，或建立新的 AI session。

Audit log 要做在哪裡？

建議放在後台的「稽核紀錄 / Audit」頁，也可以嵌在每個 conversation 或 ticket 的右側時間軸。它不是給客戶看的，而是給客服主管、營運、工程與稽核人員追查用。

  • 客服主管：檢查 AI 是否太早轉真人或錯誤回答。
  • 營運 / PM：分析哪些 SOP 不足、哪些問題最常被問。
  • 工程：debug routing、RAG 命中、tool call 失敗。
  • 稽核 / 管理者：確認誰在什麼時間做了什麼操作。

Design Choice

RAG Bot 和 Agentic Workflow 的差別

RAG bot 解決的是「回答準不準」；agentic workflow 解決的是「事情能不能被安全地完成」。兩者不是互斥，正式系統通常會同時需要。

RAG 客服

  • 適合 FAQ、SOP、產品規格、政策說明
  • 主要能力是搜尋文件並整理回答
  • 風險較低，MVP 可以很快上線
  • 找不到資料時轉真人

Agentic Workflow

  • 適合加入購物車、預約、購票、查訂單
  • 主要能力是判斷流程並呼叫 tools
  • 需要 conversation state、confirmation gate、audit log
  • 高風險操作預設轉真人或二次確認

Build Roadmap

建置順序：先做可靠的一條流程，再擴成 agent team

第一版不要追求全自動。先選一個明確 action，例如加入購物車或預約球場，跑通 LINE 對話、工具呼叫與確認流程。

Phase 1

LINE Webhook + Log

建立 Messaging API webhook、驗證 signature，將每則訊息寫入 Supabase，形成可追蹤的 conversation history。

Phase 2

Intent Router + RAG

實作 Supervisor classifier，將 FAQ 類問題 route 到 Knowledge Agent，先讓系統能穩定根據文件回答。

Phase 3

One Action Tool

先接一個 action，例如 `addToCart` 或 `createBooking`，用固定 workflow 管理缺少資訊、確認與執行。

Phase 4

Handoff + Dashboard

加入真人接手、ticket 指派、AI 停止自動回覆、客服後台與操作紀錄，讓系統可以正式營運。

Guardrails

真正的安全感來自流程，而不是 prompt 寫得很嚴格

Prompt 可以提醒 agent，但不能取代權限、狀態機、工具 schema、審計紀錄與人工接手。下面這些規則應該寫在系統層。

1

工具白名單

Agent 只能呼叫你定義好的 tools，不能直接改 database，也不要自由操作第三方網站。

2

確認閘門

加入購物車、建立預約、購票、付款、取消訂單等動作，必須先向客戶確認。

3

人工接手

退款、客訴、法律、合約、付款失敗、低信心回答，直接建立 ticket 並停止 AI 自動回覆。

4

Audit Log

每次 tool call 都要記錄 input、output、使用者確認內容、執行結果與負責 agent。

// Session-scoped handoff: only this conversation is locked to human support.
if (conversation.status === "human_active") {
  return notifyAssignedAgent(message)
}

const route = await supervisor.classify(message)

if (route.risk === "high") {
  return createHumanTicket(route.reason)
}

if (route.intent === "faq_question") {
  return knowledgeAgent.answerWithRag(message)
}

if (route.requiresAction) {
  const draft = await transactionAgent.prepare(route)
  return askForConfirmation(draft)
}

if (userConfirmedPendingAction(message)) {
  return executeApprovedTool(conversation.pendingAction)
}

Appendix

真正影響成敗的，是使用者體驗與案件管理

Agentic workflow 不只是技術架構。使用者感受到的速度、清楚度、控制感，以及內部能不能把真人接手管理好，會直接決定這套系統能不能正式營運。

附錄 A：使用者體驗與反應速度

LINE 對話最怕讓客戶等在黑洞裡。最好的體驗不是每個任務都 1 秒完成，而是 1 秒內讓使用者知道系統正在處理，並且每一步都清楚告訴他下一步是什麼。

0-1 秒
立即回饋顯示 loading animation，或先回「我先幫你查」。
1-3 秒
完成 routing用 hard rules + 輕量 classifier 判斷 FAQ、購物、預約或轉真人。
3-5 秒
給選項或結論回覆簡短結論、quick replies、可預約時段或商品選項。
5 秒以上
分段處理先告知「仍在查詢」，完成後再 push 結果，不讓客戶以為系統壞了。
  • 常見 FAQ、保固、營業時間、付款方式可以 cache。
  • 文件 embedding 必須預先建立，runtime 只做 retrieval。
  • RAG、客戶資料、商品庫存可平行查詢，不要排隊做。
  • LINE 回覆要短，複雜流程用按鈕與選項降低輸入成本。

附錄 B：真人客服需要 Ticket 系統

如果真人客服只是偶爾看 LINE OA，可以先手動處理；但只要你希望 AI 正式轉真人、指派客服、追蹤處理進度，就需要 ticket / case management。它可以先很輕量，不一定一開始就導入大型客服平台。

沒有 Ticket 有 Ticket
客服不知道哪些案件待處理。 每筆轉真人都有 open / assigned / resolved 狀態。
多位客服可能重複回覆同一位客戶。 可以指派 owner、team、priority 與 SLA。
AI 不知道何時該停止或恢復自動回覆。 `conversation.status` 控制 AI 是否可以回覆。
主管難以追查客訴、退款、付款爭議。 ticket 詳情頁保留對話、摘要、回覆與 audit timeline。
  • MVP 可先自建 `tickets` table，不必一開始買 Zendesk 或 Intercom。
  • 最小後台需要案件列表、案件詳情、客服回覆框、結案按鈕。
  • 未來若已有 CRM /客服平台，可把 `createHumanTicket` tool 改成串接外部系統。
  • 真人接手時，客戶端仍只看到 LINE OA；內部才看到指派與處理流程。
  • handoff 狀態應該綁定 conversation / session，不是讓整個 LINE OA 的 AI 客服全域停用。

建議第一版只做一個可控任務

最好的 MVP 是「LINE 問答 + 文件 RAG + 一個 action tool + 明確確認 + 真人接手」。這條路跑穩後，再擴到購票、預約、訂單查詢與更多 agent。
