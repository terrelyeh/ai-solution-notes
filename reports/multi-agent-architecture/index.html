Multi-Agent Architecture

什麼時候才需要多個 Agent

Multi-agent 不是更聰明的免費升級，而是用成本、延遲、除錯難度和更多失敗面，換取專業化、隔離與平行處理。這份筆記說明何時該拆、怎麼拆，以及如何避免一開始就過度設計。

一句話總結

先把 single orchestrator + tools 做扎實。只有當工具太多、風險邊界不同、任務需要不同 prompt / model，或專長流程已經複雜到需要獨立狀態時，才拆成 multi-agent。

什麼才算 Multi-Agent

不是：一個 Agent 用很多工具

一個 Orchestrator 搭配 RAG、cart、booking、ticket tools，仍然是 single agent + tools。

是：多個決策體

每個 agent 都有自己的 system prompt、工具子集、模型選擇、上下文合約與安全邊界。

什麼時候才該拆

觸發訊號單一 Agent 的痛點Multi-Agent 的價值
工具太多模型常選錯 tool每個 specialist 只看自己的工具集
Prompt 衝突客服語氣、交易規則、安全限制混在一起每個 agent 有自己的 system prompt
模型需求不同所有任務都用同一個模型，成本不平衡不同 agent 可用不同模型
需要平行單一路徑一步一步跑太慢Supervisor 可 fan-out 平行處理
高風險隔離交易或內部 API 能力暴露在同一個 agent高風險 agent 獨立權限與稽核

四種常見 Topology

Supervisor / Worker

一個主管 agent 負責派工、收結果、合成最終答案。

Agent as Tool

把 specialist agent 包成一個 tool，是從 single agent 演進最自然的方式。

Pipeline

固定順序，例如 Planner → Executor → Reviewer。

Handoff

把對話控制權轉交給另一個 agent，像客服分流。

推薦：先用 Agent-as-Tool

Single Orchestrator → searchKnowledgeBase tool → askTransactionSpecialist tool → Transaction Agent → searchProducts → addToCart → createBooking → createHumanTicket tool

Agent-as-tool 的好處是不用打掉 single-agent 架構。你只是把某個複雜工具升級成「內部自己會跑 agent loop 的 specialist」。

多 Agent 的主要風險

風險問題控制方式
Context loss子 agent 看不到完整脈絡使用結構化 task spec
Cost每個 agent 都可能呼叫模型限制深度、使用便宜模型
Latency多段 LLM 呼叫讓體驗變慢能平行才平行，簡單問題不要拆
Inconsistency不同 agent 可能結論不同Supervisor 對最終答案負責
Observabilitytrace 從線性變成樹記錄 parent-child agent calls

對 LINE OA 的結論

LINE OA MVP 不應該一開始就做成 multiple agents。第一版應該是 single orchestrator + tools。等 Knowledge、Transaction、Booking、Support 真的變成獨立複雜流程，再拆成 multi-agent。
